Gemba — the factory floor for agentic projects

gemba  /ˈɡem.bə/ 現場 · “the actual place” · large-scale side-of-the-desk projects: big impact, minimal cognitive load.

In lean manufacturing, gemba is the factory floor — where real work happens. A gemba walk is when leadership observes the work directly, not through reports, and leaves actionable feedback as they go. Gemba the product is built around that metaphor.

Operators running agentic software projects today hit a consistent failure pattern: planning is disconnected from execution (issues in Jira or Beads, agents in a terminal, output in git); priority is invisible (the dep graph says what can run, nothing says what should); expertise is siloed (product judgment, architecture, UX, QA, release, security — the operator can’t personally don each hat); state is fragile across git + issues + LLM sessions + sidecar artifacts with no “undo back to yesterday”; and orchestration tooling is locked to one runtime.

Gemba is a browser-based UI for walking the floor of an agentic project — seeing the work at the right grain, directing it, with a roster of configurable LLM specialists a click away for the expertise the operator doesn’t have time to personally deliver for every decision.

What this page is. A self-contained visual proof of the ratified ui-spec.md, populated against a single example project (Marker — a personal bookmark manager) so every persona, Epic state, and escalation reads as something concrete rather than abstract. Each of the five screens below is paired with its purpose and a narrative of how Mike (the operator) uses it right now, given Marker’s current state.

Main concepts

Everything below shows up on the five screens. Skim this block and the per-screen commentary will read as concrete choices against these primitives rather than introductions.

Two planes

Every project pairs exactly one WorkPlane (issue tracker) with exactly one OrchestrationPlane (agent runtime). Reference pack-ins: Beads + Gas Town. Forcing-function adaptors: Jira, LangGraph. The UI is adaptor-agnostic — no backend vocabulary leaks in.

Coaches vs Managers

Coaches are advisory (Architect, Code Reviewer, UX Expert, Documentarian, Technical Writer, Onboarder). Managers are agentic and can gate-block (Project Manager, QA, Deployment Engineer, DevOps/SRE, Security). Every persona has a TOML file, configurable prompt, and per-persona model choice.

PM panel + Gemba walks

The Project Manager is the central persona — always one click away in the PM panel. A Gemba walk is an interactive, resumable, multi-topic conversation where the PM pulls an agenda from every escalation source and you ratify / modify / reject / defer each item in turn.

Epic-granular Kanban

The Gemba is the home screen: cards are Epics, not individual stories. Each card carries readiness counts, parallel-group membership, escalation state, and token-budget posture. Drill-down reveals member WorkItems in stage order.

Values · guardrails · phases · modes

Four seed values (innovation, transparency, execution, empathy) inject at the top prompt layer for every persona. Guardrails are values with Manager-enforced blockers. Phases (building / shipping / …) and modes (unsupervised / supervised / managed) govern execution friction.

Workspace repos pair

Every project pairs <project>_prime (workspace: docs, decisions, the .gemba/ sidecar) with <project> (output: shippable code). Checkpoints snapshot both atomically — the “undo back to yesterday” is real.

Marker — the example project used below

Marker is a personal bookmark manager with web UI, CLI, HTTP API, full-text search, browser extension, and Pocket/Raindrop import. Operator: Mike. Workspace: marker. Phase: building. Mode: supervised. Adaptors: Beads + Gas Town. Personas: the 11-member CTO pack (6 Coaches + 5 Managers).

The project has 8 Epics. mk-e1 (Core API) is Done; mk-e2 (Web UI) is In Progress at 62% of its token budget with three workers active and one session going stale; mk-e3 (CLI) is Staged and parallel-safe with mk-e2; mk-e4 (Browser extension) is Next Up; mk-e5 (Full-text search) is Next Up with all members ready; mk-e6 (Pocket/Raindrop import) is Backlog with an active adaptor-degraded escalation because Pocket deprecated its v1 API; mk-e7 (PWA polish) is Backlog; mk-e8 (Team sharing) is Backlog with a Security Purview violation blocking stage. That’s the state every screen renders against.

1 · The Gemba — home / Epic Kanban

The factory-floor view. Six columns, cards are Epics, drag-to-stage as the primary action.

Purpose

The Gemba is the home screen — a six-column Epic-granular Kanban (Backlog → Next Up → Staged → In Progress → Done → Canceled) the operator opens first each session to see the state of the whole project at the right grain. Every card carries readiness counts, escalation state, parallel-group membership, and token-budget posture, so “what’s going on” is answered without drilling in. The primary action is drag-to-stage — pulling an Epic from Next Up into Staged queues it for the next Execute batch. Sidebar collapses to a 56px rail on the Board (per ui-spec §4.1) so the floor fills the viewport.

How Mike uses it — Marker right now

Mike opens Gemba to get his overview. He sees mk-e2 (Web UI) in progress at 62% of its epic token budget with parallel-group glyph A; mk-e3 (CLI) sitting in Staged, also group A and safe to run alongside; mk-e6 (Pocket/Raindrop import) in Backlog wearing a red escalation dot because Pocket deprecated their v1 API overnight; and mk-e8 (Team sharing) in Backlog with a yellow purview-violation dot — Security flagged auth-surface work.

He’s mid-drag on mk-e4 (Browser extension), pulling it from Next Up toward a dashed ghost slot in Staged. The ghost copy honestly reads “Drop to Stage · nonce-gated · supervised mode” so the mutation surface doesn’t lie about the confirmation path. Before committing the drop he could Cmd-K to any other surface, or open the PM panel at the bottom and click Recommend order to have the PM reason about whether mk-e4 belongs staged yet given the group-A workload and mk-e6’s escalation. The top-bar dual budget gauge splits agent spend ($18.42 today) from advisor spend ($1.51 today) so he always knows which pool he’s pulling from.

2 · Gemba walk — active decisioning

Agenda-left, chat-right. The PM frames an adaptor-degraded escalation; the operator ratifies.

Purpose

The Gemba walk is the operator’s interactive decisioning surface — an interactive multi-topic conversation with the PM, populated on start from every escalation source in the workspace (open EscalationRequests, HITL questions, recently-filed beads, recently-closed work needing retro, and user-added topics). The walk is resumable across days and auditable turn-by-turn; on end, Documentarian writes a summary artifact. Each agenda item cycles through the ratified verbs — Ratify / Modify / Reject / Defer — with the PM consulting other personas inline for perspective. The PM panel doubles as the walk’s chat surface, so “asking the PM” and “working the list” are the same gesture.

How Mike uses it — Marker right now

Mike runs a Gemba walk after Security’s Purview flag on mk-e8 lands alongside the overnight Pocket API escalation. The PM has auto-populated five agenda items: two already decided (a stale_session on mk-e2.4 — restart worker; and a HITL question about sprint placement for mk-e5), one active (the mk-e6 Pocket API v1 deprecation, currently highlighted), and two queued (the Security Purview violation on mk-e8 and a user-added “review mk-e2 EOD”).

The chat pane shows the PM framing three options for the Pocket problem — (a) update mk-e6 to v2, (b) pivot to Raindrop-only, (c) defer mk-e6 until v1 is fully retired — with inline quote-style perspectives from Architect (design impact), Deployment Engineer (dormant-Purview note on release surface), and Documentarian (user-docs implications). Two more perspectives (QA, Security) are collapsed under an “expand all” affordance. Mike hovers Ratify (a) — the navy primary CTA — and the tooltip confirms nonce-gating under supervised mode. One click applies the mutation through the same API any client uses. The walk has been running 6 minutes; accumulated cost is $0.04 in the PM panel header.

3 · Epic drawer — drill into mk-e2

Right-side overlay on a dimmed Board. Mini-kanban sliced by stage swimlanes.

Purpose

The Epic drawer is where the operator goes when an Epic’s Board card doesn’t answer their question — when they need to see which members are in flight, where the friction actually is, and whether to intervene. It overlays the Gemba at 40% of viewport width and dims the Board behind it, so context is preserved on dismiss. The body renders a four-column mini-kanban (Ready / In Progress / Review / Done) sliced by stage swimlanes; the header carries the state pill, priority badge, parallel-group glyph, and the actions toolbar (Stage / Start / Complete / Defer / Split / Merge / Add member / Open in graph) — with transition verbs dimmed when their preconditions aren’t met.

How Mike uses it — Marker right now

Mike double-clicks mk-e2 on the Board to see what’s actually happening inside the Web UI Epic. The drawer opens: 7 members split across three stages. Stage 0 (Scaffolding) has mk-e2.1 and mk-e2.2 both Done; Stage 1 (Feature surfaces) has three in-progress workers — obsidian on mk-e2.3 (BookmarkList), jasper on mk-e2.4 (BookmarkDetail), onyx on mk-e2.5 (Search bar); Stage 2 (Polish) has mk-e2.6 and mk-e2.7 ready.

mk-e2.4 renders with an amber-outlined stale marker — the worker’s gone quiet for 14 minutes. Mike hovers the stale dot and a dark popover surfaces the three Witness-triaged options: Restart worker (primary), Peek session (open the live transcript), Escalate to Gemba walk (add it to the PM’s agenda). The Complete button in the actions toolbar is dimmed because mk-e2 isn’t Review-complete yet — the DoD guardrail is honest about that. Metadata chips on the tab bar (3 comments · 2 evidence · 8 events) preview drill-down depth without cluttering the primary surface.

4 · Plan view — staging surface

Pick what runs next and in what parallel group. Drag between clusters to merge scopes.

Purpose

Plan is the batch-launch surface — the place the operator goes to decide what runs next, in what parallel groups, and at what projected spend before dispatching the wave. The primary action is Execute all Staged, with a sub-badge quantifying cost and token envelope up front so nothing launches by accident. Staged Epics cluster by parallel-safety group (blue outlines for verified, tan-dashed for “eligible to merge,” red-X for “not parallel-safe” with an explanation). Drag an Epic between clusters to merge or separate scopes; the sidebar is expanded here because Plan isn’t the Board.

How Mike uses it — Marker right now

Mike wants to plan the next wave. Typing /plan groups Staged + eligible Next-Up Epics by parallel-safety. Group A (verified) contains mk-e2 (Web UI, running now — the inset “running” pill shows live spend $0.41, 62% budget) and mk-e3 (CLI, Staged, 4/4 ready, ~60k est, $1.20). Group B (“eligible to merge into A”) contains mk-e5 (Full-text search, 3/3 ready, ~42k, $0.84). Below those, a red-X cluster flags mk-e4 (Browser extension) as not parallel-safe with A because its file signature overlaps cmd/marker/* with mk-e3.

Mike drags mk-e5 out of Group B into Group A. A ghost row follows the cursor with a Join Group A affirmation chip; the Group B source row dims to “dragging to group A…”. The cost-preview row re-projects on drop. He clicks Execute all Staged to dispatch roughly 142k tokens of work — ~$2.40, ~3h wall-clock — with the stacked share-of-sprint-budget bar showing how much of the day’s envelope this consumes. mk-e7 surfaces under “Next candidates” with a Stage when ready — needs DoD review guidance, enforcing the “Every Epic requires DoD before Start” guardrail.

5 · Persona roster — Coaches & Managers

11 configured personas. The Project Manager card is expanded, showing PPPP + skills + budget.

Purpose

The persona roster is where the operator tunes the workspace’s expertise layer — the Coaches and Managers the PM and the operator both call on. It renders as a grid of 4-wide persona cards with variety badges (purple for Coach, navy for Manager per §1.2 + §3.4). Clicking a card expands a drawer below with its four PPPP axes ratified by gm-9rv / gm-uf7: Personality, Perspective, Purview, and per-persona Skills. Each persona is a TOML file; Edit opens a modal editor over .gemba/personas/<role>.toml. Packs bundle personas + skills + default values + guardrails + adaptor preferences.

How Mike uses it — Marker right now

Mike reviews the roster to tune the Marker workspace. 6 Coaches + 5 Managers, all toggled on; advisor spend $1.51 today against the $20 cap. The CTO pack label reads “active · installed 3d ago” at the top. The Project Manager card is expanded — navy-outlined, caret down, connected by an arrow to the drawer below. The drawer surfaces PM’s PPPP: Personality “Precise and economical — laconic,” Perspective “Coordination, dep-graph awareness, escalation triage,” Purview explicitly rendered as (none — PM is orchestrator, not domain-owner) with an INVARIANT reminder that PM never gates state transitions.

The right column lists 12 opt-in Skills (epic_order, parallel_safety, scope_trim, what_remains, add_to_backlog, change_this, dispatch_readiness, etc.), a Budget Policy row ($0.25/invocation · $20/day · 3.7% cap-pressure derived from today’s $0.73 PM spend), and Context Providers (5 of 7 opted in — code_summary and call_graph available but off, consistent with the per-persona opt-in pattern). Glancing down the roster, Mike can see Security has been consulted recently (~$0.09 today) on the mk-e8 Purview flag, and Deployment Engineer sits idle at $0.00 because Marker hasn’t entered its shipping phase yet. He clicks Edit on the PM card — the tooltip confirms “Open TOML editor · .gemba/personas/pm.toml” — to tighten the system prompt on escalation-triage framing.
